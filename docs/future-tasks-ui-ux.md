# 今後課題（フロント分割とUX改善）

## 1. このドキュメントの目的
本ドキュメントは、現行実装をすぐ変更せずに、次フェーズで取り組むべき課題を合意するための整理資料です。

- 対象1: 画面コンテンツ単位でのフロント分割方針
- 対象2: 旧画面と比較したユーザーフレンドリー化のデザイン課題
- 前提: 今回は実装しない

## 2. フロントをコンテンツごとに分けるべきか
### 2.1 結論
分けたほうが良いです。  
ただし、いきなり細分化せず、画面の責務境界が明確な単位から段階的に分割する方が安全です。

### 2.2 理由（メリット）
- 変更影響範囲が小さくなるため、改修時の事故を減らせる。
- 表示ロジックとデータ取得ロジックを分離しやすくなり、レビューしやすい。
- 再利用しやすい部品（検索フォーム、ログカード、詳細ヘッダ）が作れる。
- 将来の機能追加（ステータス絞り込み、並び替え、ページング）に対応しやすい。

### 2.3 懸念（めんどくさくなるか）
結論として「少し増えるが、運用上は楽になる」です。

増える作業:
- Propsの受け渡し設計
- ファイル数の増加
- 命名ルールの統一

減る作業:
- 1ファイル肥大化に伴う読解コスト
- 不具合調査時の切り分けコスト
- 似たUIの重複修正コスト

### 2.4 複雑化を防ぐ運用ルール（実装時の条件）
- 1コンポーネント1責務に限定する（検索、一覧、詳細を混在させない）。
- 状態は3区分で管理する。
  - サーバー状態: TanStack Query
  - 画面ローカル状態: 入力中フォーム、開閉状態
  - 共有状態: 認証情報など
- Propsドリルが深くなる場合は、ViewModel整形を上位で完結させる。
- 画面分割の最小単位は以下を推奨する。
  - TranscribeFiltersPanel
  - TranscribeLogList
  - TranscribeDetailHeader
  - TranscribeInputPointList

## 3. 旧画面比較で見えたUX課題
旧画面スクリーンショットの強みを基準に、現行画面の改善課題を整理します。

### 3.1 現行で不足している体験
- 一覧カードの情報密度と視認性が弱い（日時、状態、発信元の視線誘導が弱い）。
- フィルタ領域の操作感が弱い（入力群と検索アクションの区切りが目立たない）。
- 状態の意味が色で直感的に伝わりにくい（未対応、対応完了など）。
- 詳細の情報グルーピングは改善済みだが、優先情報の強調順はまだ最適化余地がある。
- 音声再生コンテンツの文脈（どの入力ポイントに対応するか）が一目で分かりにくい。

### 3.2 改善テーマ（デザイン）
#### テーマA: 一覧の可読性強化
- カード内の文字階層を明確化（主情報: 日時、補助情報: 状態・発信元・通話時間）。
- 選択中カードのハイライトを色＋太さ＋余白で明確化。
- 状態表示はチップ化し、意味ごとに色を固定する。

#### テーマB: フィルタ操作の迷い削減
- フィルタブロックを視覚的に1ユニット化し、検索ボタンを常に同位置に固定。
- 日付入力の関係性（開始から終了）を補助文で明示。
- 検索実行後の適用条件を小さく表示し、現在の抽出条件を可視化。

#### テーマC: 詳細の意思決定速度向上
- 詳細上部は「対応判断に必要な項目」を最上段に集約。
- 録音、ユーザー状態、通話時間を同一視線上に配置。
- 入力ポイントはカード化し、質問ID、入力結果、区間を固定順で表示。

#### テーマD: 反応性と安心感
- 読み込み中はスケルトンまたは占有プレースホルダで待ち時間を説明。
- 空状態、エラー状態を文言と再試行導線で統一。
- 小画面での2カラム崩れ時に、主要操作が最上部に残るように配置する。

## 4. 優先度つき課題バックログ
### 優先度High（先に着手）
- 一覧カードの情報階層ルールを定義。
- 状態チップの色ルールを定義。
- フィルタ領域のUIルール（ラベル、余白、ボタン位置）を定義。

### 優先度Middle
- 詳細ヘッダの情報優先順位を再設計。
- 入力ポイント表示フォーマットを統一。
- ローディング、空状態、エラー状態のUIパターンを統一。

### 優先度Low
- モーション（選択時、読み込み完了時）の最小導入。
- アクセシビリティ強化（フォーカスリング、コントラスト最適化）。

## 5. 完了条件（次フェーズでの受け入れ基準）
- 初見ユーザーが3クリック以内で「対象ログの特定」と「詳細確認」を完了できる。
- 一覧カードの情報優先度がチーム内で一致し、実装差異が出ない。
- フィルタの適用条件をユーザーが常に把握できる。
- 空状態、読み込み、エラー時にユーザーが次の行動を判断できる。

## 6. 実施順（提案）
1. UIルール定義（配色、余白、タイポ、状態チップ）
2. 画面分割方針確定（コンポーネント境界だけ先に決める）
3. 一覧とフィルタのデザイン改善
4. 詳細と入力ポイントのデザイン改善
5. 最後にアクセシビリティとモーションを最小追加
